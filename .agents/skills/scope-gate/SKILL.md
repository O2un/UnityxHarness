# Scope Gate

## 역할

설계·구현에 **들어가기 전에** 요청(또는 이미 작성된 PRD)의 크기를 판단한다. 크면 착수를 막고 **의존성 순서가 흐르는 PRD 분해안**을 제시한다. 판정만 하고 **자동으로 작업을 시작하지 않는다** — 사용자 승인 뒤에 후속 단계로 넘어간다.

사람의 규모 눈대중은 실제보다 작게 잡히기도 한다. 착수 전 한 번 더 걸러 "큰 걸 통째로 태우는" 비용 사고를 막는 게이트다.

## 호출 경로 (두 갈래)

- **자동**: `unity-dev-orchestrator` 진입 시 설계 착수 전에 규모를 판단한다.
- **수동**: 사용자가 `/scope-gate`로 직접 호출한다. `$ARGUMENTS`가 PRD 파일 경로면 **검증 모드**, 비어 있으면 **현재 대화의 요청**을 판단 대상으로 삼는다.

## 동작 절차

1. **대상 확정** — `$ARGUMENTS`가 파일 경로면 그 PRD를 Read로 읽어 검증 모드. 비어 있으면 현재 요청을 판단 대상으로 삼는다.
2. **신호 평가** — 아래 판단 기준을 하나씩 대조한다. 걸린 기준을 근거로 기록한다.
3. **판정** — 기준 **2개 이상**에 걸리면 `분리 필요`, 아니면 `한 번에 진행`.
4. **출력** — 판정과 걸린 근거를 제시한다. `분리 필요`면 분해안 표를 함께 낸다.
5. **승인 대기** — `분리 필요`면 "이 경계로 진행할까요, 조정할까요?"를 묻고 **승인 전 착수 금지**. 승인 시 `prd` 분리 모드로 PRD를 하나씩 생성하도록 넘긴다.

## 판단 기준 ("큰 요청" 신호 — 2개 이상이면 분리 필요)

- **여러 시스템에 걸침**: 독립적으로 검증 가능한 시스템/기능이 2개 이상 얽혀 있다.
- **다단계 의존 흐름**: 한 부분의 산출물(이벤트·인터페이스·데이터)이 다음 부분의 입력이 되는 순차 의존이 존재한다.
- **레이어 다수 신설**: Manager/Module/Service/View 계열을 여러 벌 새로 만든다.
- **신규 파일 다수**: 새로 생길 `.cs`가 대략 5개를 넘길 것으로 보인다.
- **요구가 3마디 이상**: "A 하고 B 하고 C까지" 형태로 이어진다.

## 출력 계약

- 판정 결과를 먼저 명시한다: `한 번에 진행` 또는 `분리 필요`.
- `분리 필요`일 때 아래 분해안 표를 낸다. 각 행이 PRD 후보 1건이고, 하나의 모듈 경계에 대응한다. 앞 후보의 **출력(Provides)**이 다음 후보의 **입력(Depends On)**이 되는 사슬을 이룬다.

  | 순서 | PRD 후보 | 입력(Depends On) | 출력(Provides) |
  |---|---|---|---|
  | 1 | … | (선행 조건) | (다음이 받을 산출) |
  | 2 | … | 1의 출력 | … |

- 후속 안내: "이 경계로 진행할까요, 조정할까요?"를 묻는다. 승인 전 설계·구현·PRD 생성 착수 금지.

## 수동 검증 모드 (인자에 PRD 경로)

- 대상 PRD를 읽어 그 안에 독립 검증 가능한 시스템이 몇 개 섞였는지, 판단 기준에 몇 개 걸리는지 리포트한다.
- 하나로 뭉쳐 있고 기준 2개 이상에 걸리면 "N개로 나누는 것을 권장"과 함께 위 분해안 표를 제시한다.

## 금지

- 이 스킬은 파일을 **생성·수정하지 않는다**. 읽기·판단·제안까지만 한다.
- 판정 후 자동으로 설계·구현·PRD 생성을 시작하지 않는다. 사용자 승인을 기다린다.
- 규모가 "작다"고 판정된 요청은 게이트만 통과시키고 기존 흐름에 맡긴다 — 임의로 개입하지 않는다.
- 비용·토큰을 정량 측정하지 않는다. 파일 수·시스템 수 등 구조적 신호로만 판단한다.
