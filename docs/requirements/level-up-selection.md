# LevelUpSelection PRD

> 근거: `docs/spec/LevelUpSelection.md` · 경험치 루프의 끝(PRD 3). 입력은 PRD 2(`ExperienceSystem`)의 `LevelUpEvent`. 이 시스템으로 처치 보상 루프가 한 바퀴 닫힌다.

## Overview

`LevelUpEvent`를 받아 게임을 멈추고, 능력 후보 세 개를 버튼으로 보여준 뒤 하나를 고르면 능력을 적용하고 게임을 재개하는 선택 UI 계층. 경험치 루프의 마지막 단계로, 경험치 누적·레벨 판정(PRD 2)에는 관여하지 않고 "레벨업 발생 → 선택 → 적용 → 재개"만 책임진다.

UI는 **View / ViewModel(VM)** 로 구성한다. View(MonoBehaviour)는 표시·바인딩만 담당하고, 선택 상태·후보 목록·버튼 활성 여부 등 표시용 상태는 VM이 보유한다. Context는 `LevelUpEvent` 구독·일시정지·능력 적용·재개 흐름을 조율하고 VM을 통해 View와 통신한다.

## Goals

- `LevelUpEvent`를 구독해 레벨업 순간 게임을 멈춘다 (표현 계층 결정: `Time.timeScale` 또는 상태 전환)
- 능력 정의 목록(데이터)에서 후보 세 개를 뽑아 세 버튼으로 표시한다
- 버튼 하나를 고르면 해당 능력을 플레이어에게 적용하고 게임을 재개한다
- 능력 후보를 기존 스킬 데이터(`SkillDefinitionSO`, 4A-3c12)로 구성해 대미지·쿨타임·범위·투사체 수 등 여러 방향을 담는다
- UI를 View/VM으로 분리하고, 구독·일시정지·능력 적용은 Context에 두어 `ExperienceModule` 내부를 참조하지 않는다

## Out of Scope

- 경험치 누적·레벨 판정·`LevelUpEvent` 발행 (PRD 2 `ExperienceModule` 소관)
- 능력 후보 데이터 자체의 신규 설계 — 기존 스킬 데이터(`SkillDefinitionSO`) 재사용
- 후보 추첨 확률·가중치·등급(rarity) 테이블 — 이번엔 단순 무작위 추출
- 리롤·스킵·후보 잠금 등 선택 편의 기능
- 선택 UI 연출(애니메이션·사운드) 및 HUD 게이지(PRD 2 구독 측 책임)

## Technical Requirements

### 배치

- 모든 구성요소는 **`10_ProjectA`(프로젝트 A 전용)** 아래에 둔다. (`SkillModule`·`SkillDefinitionSO` 등 기존 공통 인프라는 `00_CommonFramework`에서 재사용)
- 구독·일시정지·능력 적용은 **Context**, 표시·바인딩은 **View/VM** 계층이다. 순수 로직 Module이 아니라 Unity 표현 계층(UI·`Time.timeScale`)에 접점을 갖는 것이 정상이다.

### 구성요소

| 구성요소 | 계층 | 책임 |
|---|---|---|
| `LevelUpSelectionContext` | Context | `LevelUpEvent` 구독 → 일시정지 → 후보 추출 → VM 갱신 → 선택 결과 수신 → 능력 적용 → 재개. 다중 레벨업 큐잉 |
| `LevelUpSelectionViewModel` | VM | 후보 목록·버튼 표시 상태 보유, 버튼 클릭을 선택 이벤트로 발행, View와 Context 사이 바인딩 매개 |
| `LevelUpSelectionView` | View (MonoBehaviour) | 패널 표시/숨김, 세 버튼 바인딩, VM 상태를 화면에 반영 |
| 능력 후보 추출 | Context 내부 로직 | 능력 정의 목록에서 무작위 3개 추출 (보유 여부 무관) |
| 능력 정의 목록 | Data | 후보 풀 = `SkillDefinitionSO` 목록 (기존 스킬 데이터 재사용) |

### 입력 (상류 연결)

- `LevelUpEvent`(PRD 2)를 R3로 구독한다. payload의 **결과 정보(새 레벨 등)만** 사용한다.
- `ExperienceModule` 내부를 직접 참조하지 않는다 — 단방향.

### 일시정지

- 레벨업 수신 시 게임을 멈춘다. 멈춤 방식은 표현 계층이 결정한다 (`Time.timeScale = 0` 또는 상태 전환).
- `ExperienceModule`은 일시정지에 관여하지 않는다.
- `Time.timeScale = 0` 방식일 경우 선택 UI 입력이 정지 중에도 동작해야 한다 (UI는 unscaled).

### 다중 레벨 상승 처리 (큐잉)

- 기본적으로 다중 레벨업은 발생하지 않는다. 다만 한 번에 2레벨 이상 오르면 `LevelUpEvent`가 레벨마다 1회씩 온다(PRD 2).
- **큐잉으로 처리한다**: 들어온 이벤트 수만큼 선택 창을 연달아 띄운다. 하나를 고르면 큐에 남은 다음 레벨업 선택 창을 이어서 띄우고, 큐가 빌 때 게임을 재개한다.

### 능력 선택 & 적용

- 능력 정의 목록에서 후보 **세 개**를 뽑아 세 버튼으로 표시한다. 후보는 하드코딩하지 않고 데이터 목록에서 뽑는다.
- 버튼 클릭 시 해당 능력을 플레이어에게 적용한 뒤(큐가 비면) 게임을 재개한다.
- **새 스킬 습득 + 레벨 덮어쓰기**: 선택한 능력을 플레이어의 활성 스킬 집합(현재 `PlayerActor`가 보유한 `SkillModule` 구성)에 적용한다.
  - 플레이어가 아직 보유하지 않은 스킬이면 **새로 습득**한다.
  - 이미 보유한 스킬이면 **더 높은 레벨로 덮어쓴다** (선택된 능력 정의의 레벨이 기존보다 높을 때 교체). 스택 병합이 아니라 레벨 교체 방식이다.
- **`SkillModule`을 런타임 습득/덮어쓰기가 가능하도록 수정한다.** 현재는 생성자에서 `ISkillDefinition[]`를 받아 고정이므로, 다음을 노출하도록 변경한다:
  - 미보유 스킬을 **추가(습득)** 하는 API
  - 이미 보유한 스킬을 **더 높은 레벨의 정의로 교체(덮어쓰기)** 하는 API
  - 위 두 동작을 하나로 합친 "적용" API로 두어도 된다 (미보유면 추가, 보유면 레벨 비교 후 교체).
- 구체 API 시그니처는 설계 단계에서 확정한다.

### 데이터 (능력 후보)

- 후보 풀은 기존 스킬 데이터(`SkillDefinitionSO`)로 구성한다. 대미지·쿨타임·범위·투사체 수 등 서로 다른 방향의 능력을 담는다.
- 능력 후보 목록을 인스펙터에서 데이터로 주입한다. Context/View/VM에 후보를 하드코딩하지 않는다.
- **후보 부족 시**: 추출 가능한 후보가 3개 미만이면 **있는 것만 표시**한다 (빈 슬롯·더미 후보로 채우지 않는다).

### UI 캔버스

- 선택 UI는 **기존 캔버스 하나를 유지**해 그 위에 올린다. 전용 캔버스를 새로 두지 않는다.

### 계약 (루프 닫힘)

- 출력은 "능력 적용 후 게임 재개" — 루프가 재시작된다.
- 구체 적·아이템 타입을 참조하지 않는다.
- R3 구독은 모두 `AddTo`로 lifecycle에 바인딩한다. R3 사용 범위는 `ReactiveProperty`·`Subject`·`Subscribe`·`AddTo`로 한정.

## Acceptance Criteria

- [ ] `LevelUpEvent` 수신 시 게임이 멈춘다 (`Time.timeScale` 또는 상태 전환)
- [ ] 능력 후보 세 개가 데이터 목록에서 뽑혀 세 버튼으로 표시된다 (하드코딩 없음)
- [ ] 버튼 하나를 고르면 해당 능력이 플레이어에게 적용되고 게임이 재개된다
- [ ] 미보유 스킬은 새로 습득되고, 보유 스킬은 더 높은 레벨로 덮어써진다 (스택 병합 아님)
- [ ] `SkillModule`이 런타임 습득/덮어쓰기 API를 노출하도록 수정된다 (생성자 고정 → 런타임 적용 가능)
- [ ] 다중 레벨업 시 선택 창이 큐잉되어 레벨 수만큼 연달아 뜨고, 큐가 비면 재개된다
- [ ] 후보가 3개 미만이면 있는 것만 표시된다
- [ ] 선택 UI가 기존 캔버스 하나 위에 표시된다 (전용 캔버스 없음)
- [ ] UI가 View/VM으로 분리되고, 구독·일시정지·능력 적용은 Context에 있으며 `ExperienceModule` 내부를 참조하지 않는다
- [ ] `LevelUpEvent` payload의 결과 정보(새 레벨 등)만 사용한다 (단방향)
- [ ] 구체 적·아이템 타입 참조가 없다
- [ ] R3 구독이 모두 `AddTo`로 lifecycle에 바인딩된다
