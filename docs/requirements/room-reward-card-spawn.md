# 룸 보상 카드 스폰 & 선택 (RoomRewardCard)

> 분해 3/4 — `item-upgrade-system.md`의 "룸 진행 연동 · View · 선택 UX" 축.
> **선행 의존: `player-runtime-stat-refresh.md`(1/4), `upgrade-card-data-inventory.md`(2/4).** 이 PRD는 이미 만들어진 추첨·획득 로직에 룸 클리어 트리거와 씬 상호작용을 잇는다.

## Overview

룸을 클리어하면 문 앞에 보상 카드 2장이 나타나고, 플레이어가 그중 하나에 다가가 상호작용하면 획득이 확정되고 나머지 후보가 회수된다. 현재 룸 진행은 `RoomModule`이 클리어를 판정해 `OnRoomCleared`를 쏘고 `RoomDoorView`가 문을 여는 데서 끝난다. 이 기능은 그 사이에 "선택 1회"라는 성장 스텝을 끼워 넣어, 반복되는 룸 클리어에 누적 성장 동기를 만든다. 이 PRD가 끝나면 앞의 두 PRD가 남긴 임시 개발 하네스는 모두 제거된다.

## Goals

- `IRoomProgression.OnRoomCleared`를 구독해 카드 후보 2장 추첨을 트리거한다.
- 문 앞 기준 Transform 하나(`_rewardSpawnPoint`) 기준 좌우 오프셋에 카드 2장을 스폰한다.
- 카드에 접근하면 상호작용 프롬프트를 표시하고, 벗어나면 감춘다.
- 상호작용 시 해당 카드를 획득 확정하고 **두 후보를 모두 회수**한다.
- 룸당 획득은 1회로 제한하고, 슬롯 부족 시 후보를 유지한 채 안내만 표시한다.
- 분해 1/4·2/4의 임시 개발 하네스를 제거한다.

## Out of Scope

- 패시브 스킬 카드의 실제 효과 발동(크리티컬·유도 미사일) — 분해 4/4.
- 슬롯이 가득 찼을 때의 교체 UI. 후보를 유지하고 안내만 표시한다.
- 카드 리롤·스킵·재선택.
- 카드 아트·아이콘 정식 에셋. 플레이스홀더 스프라이트로 진행한다.
- 인벤토리 화면 UI(획득 목록 표시).
- 카드 스폰·회수 연출(등장 애니메이션·파티클). 표시/비표시로 충분하다.
- 획득 카드의 저장/세션 간 영속화.
- 자동화 테스트 검증 — 테스트 어셈블리(asmdef) 도입 전까지 검증하지 않는다.

## Technical Requirements

### 배치

- 신규 코드는 `Assets/20_ProejctB/01_Scripts/Reward/` 아래, 분해 2/4가 만든 폴더에 함께 둔다.
- 공통 `00_CommonFramework`는 수정하지 않는다.

### 진행 연동

- 룸 보상 진행을 담당하는 클래스가 `IRoomProgression.OnRoomCleared`를 구독해 후보 추첨을 시작하고, `OnRoomEntered`를 구독해 룸당 1회 플래그를 초기화한다.
- 룸당 1회 제한은 **상태 플래그**로 판정한다. 카드 오브젝트를 지우는 것에 의존하지 않는다(`RoomDoorView`가 콜라이더를 껐다 켜는 대신 `_isOpen` 플래그로 막는 것과 같은 이유 — 이미 범위 안에 서 있는 경우와 타이밍 엇갈림을 피한다).
- 진행 로직은 **Unity API 비의존 순수 C#**으로 두고, 배치·시각 표현은 View에 맡긴다. 이미 분해 2/4가 만든 추첨·획득 로직을 재사용하고 여기서 다시 구현하지 않는다.
- DI: `ProjectBSceneScope`에 싱글턴 등록한다. 룸을 넘어 획득이 누적돼야 하므로 `RoomSceneScope`가 아니다.

### View

- 카드 View는 `RoomDoorView`와 **동일한 패턴**을 따른다:
  - `[RequireComponent(typeof(Collider2D))]`, 트리거 `OnTriggerEnter2D`/`OnTriggerExit2D`에서 `LayerMask` 비트 비교로 플레이어 판정.
  - `IInputReader.IsAttackPressed`를 **상시 구독**하고 핸들러 안에서 상태를 검사한다. 범위 진입/이탈마다 구독을 붙였다 떼지 않는다.
  - 프롬프트 루트를 `SetActive`로 토글한다.
  - 외부 알림은 R3 `Subject`로 노출한다(`OnSelectRequested`).
  - `CompositeDisposable`을 `OnDestroy`에서 `Dispose`한다.
- 카드는 플레이스홀더 스프라이트(단색/기본)로 표시한다. `IUpgradeCardData.DisplayName`을 텍스트로 띄우는 것은 선택 사항이며, 어떤 카드인지 구분만 되면 된다.

### 스폰 위치와 인스턴스화

- 스폰 기준점은 `RoomSceneScope`가 직렬화 필드로 제공하는 **Transform 하나**(`_rewardSpawnPoint`)다. 후보 2장은 그 지점 기준 좌우 오프셋으로 배치한다. 지점마다 오브젝트를 두지 않는다.
  - `RoomSceneScope`가 `_playerSpawnPoint`를 다루는 방식(널 검사 후 `Debug.LogError`, 위치를 프로퍼티로 노출)을 그대로 따른다.
  - 미할당 시 에러 로그를 남기고 보상 스폰만 건너뛴다. 룸 진행 자체는 계속되어야 한다.
- `GameObject.Find` 등 씬 탐색 금지. `Resources.Load` 금지.
- 프리팹 인스턴스화는 `IPoolService`(`PoolManager`)를 사용한다. `Player2DActor.FireProjectile`이 `IsRegistered` 확인 후 `Register` → `GetHandle<T>` → `Get()`을 쓰는 흐름을 따른다. 카드 프리팹은 `IPoolable`을 구현해 회수 시 상태를 초기화한다.
- 카드 View는 룸 씬에서 생성되므로 `RoomSceneScope`의 빌드 콜백 경로에서 `resolver.InjectGameObject`로 주입한다. `RoomSceneScope`가 `_doors`를 `BindDoors`에서 주입·구독 연결하는 방식을 참고한다.

### 선택 처리

- 카드 상호작용 → 진행 로직이 분해 2/4의 획득 API 호출 → 결과에 따라 분기:
  - 확정: 선택 완료 신호 발행 → **두 후보 모두 회수**(선택된 것도 사라진다) → 룸당 1회 플래그 설정.
  - 슬롯 부족: 후보 유지 → 안내 표시. 재시도 가능하며 룸당 1회 플래그를 설정하지 않는다.
- 룸 전환 시 남아 있는 후보는 룸 씬 언로드와 함께 정리된다. 풀 인스턴스가 씬과 함께 파괴되지 않도록 등록 루트를 확인하고, 필요하면 룸 이탈 시 명시적으로 회수한다.
- 슬롯 부족 안내는 프롬프트 텍스트 교체 또는 별도 안내 루트 `SetActive`로 처리한다. 별도 UI 시스템을 도입하지 않는다.

### 정리

- 분해 1/4의 `Player/Dev/` 스탯 하네스와 분해 2/4의 획득 하네스를 삭제한다.
- 기존 `Room/Dev/RoomSpawnTestTrigger`는 이 PRD의 범위가 아니므로 건드리지 않는다.

## Acceptance Criteria

- [ ] 룸의 적을 모두 처치하면 문이 열리는 것과 함께 문 앞에 카드 2장이 나타난다.
- [ ] 카드는 서로 다른 종류이며, 룸마다 다시 추첨된다.
- [ ] 카드 범위 안에 들어가면 상호작용 프롬프트가 표시되고, 범위를 벗어나면 사라진다.
- [ ] 카드가 나타나기 전(전투 중)에는 카드 위치에 서 있어도 프롬프트가 뜨지 않는다.
- [ ] 한 카드를 선택하면 **두 후보가 모두 사라진다** (선택된 것은 획득, 나머지는 회수).
- [ ] 선택 후 다시 상호작용해도 추가 획득이 일어나지 않는다 (룸당 1회).
- [ ] 선택한 카드의 효과가 **획득 즉시** 반영된다 (예: 이동 강화 선택 시 그 자리에서 빨라진다).
- [ ] 카드 6장을 획득한 뒤 선택하면 후보가 사라지지 않고 슬롯 부족 안내가 표시된다.
- [ ] 슬롯 부족 안내 후에도 문으로 이동해 다음 룸으로 넘어갈 수 있다.
- [ ] `_rewardSpawnPoint`가 비어 있는 룸에서는 에러 로그만 남고 룸 진행(문 열림·전환)은 정상 동작한다.
- [ ] 룸을 전환해도 이전 룸에서 획득한 강화가 유지된다.
- [ ] 룸을 전환하면 이전 룸의 카드 오브젝트가 남지 않는다.
- [ ] Play 모드 진입부터 2개 룸 클리어까지 콘솔 에러가 없다.

## Open Questions

- 좌우 오프셋 거리와 카드 콜라이더 크기 미정. 두 카드의 상호작용 범위가 겹치면 어느 쪽이 선택되는지 모호해지므로, 겹치지 않는 값으로 잡는다.
- 카드가 스폰되는 시점이 문이 열리는 시점과 동일해도 되는지, 아니면 약간의 지연이 필요한지 — 연출 판단.
- 룸 씬(`RoomData_1`/`RoomData_2`)에 `_rewardSpawnPoint` Transform 추가는 **씬 변경이므로 별도 승인이 필요하다.**
- 카드 프리팹 생성 및 플레이스홀더 스프라이트 배치도 **에셋 변경이므로 별도 승인이 필요하다.**
