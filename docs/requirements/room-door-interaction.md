# 문 상호작용 (room-door-interaction)

> room-system PRD 분해 4/4. 선행 PRD `docs/requirements/room-progression.md`의 `IRoomProgression` 계약(전환 진입점 `RequestTransition(string destinationId)`, 룸 클리어 신호 `OnRoomCleared`)과 룸 클리어 상태 판정을 **그대로 소비하고 재정의하지 않는다.** 진행 판단·전환 순서·중복 입력 무시 정책은 3/4 소관이다.

## Overview

룸 클리어 후 플레이어가 다음 룸으로 넘어가는 지점이 되는 문 View. 클리어 전에는 존재만 하고 아무 반응도 하지 않으며, 클리어 후 플레이어가 문 범위에 들어오면 문 자신이 소유한 월드 스페이스 프롬프트를 띄우고, 그 상태에서 공격 키를 누르면 자신의 목적지 식별자를 실어 전환 신호를 발행한다. 문은 진행 가능 여부를 스스로 판단하지 않고 신호만 전달한다.

## Goals

- `Collider2D`(`isTrigger`) 기반으로 플레이어의 문 범위 진입·이탈을 판정한다
- 룸 클리어 전에는 문을 완전히 비활성 상태로 두어 프롬프트도 신호도 발생하지 않게 한다
- 문 View가 소유한 월드 스페이스 프롬프트 UI를 범위 안 + 열린 상태에서만 표시한다
- 공통 `IInputReader.IsAttackPressed`를 구독해 범위 안 입력만 전환 신호로 바꾸고 범위 밖 입력은 무시한다
- 문이 자신의 목적지 식별자를 들고 R3 신호로 전달해, 한 룸에 문이 여러 개여도 성립하는 계약을 만든다
- 문에서 진행 판단 로직을 배제하고 신호 전달만 담당하게 한다

## Out of Scope

- 룸 전환 실행·전환 순서·페이드·플레이어 배치 — 3/4 `RoomModule` 소관. 문은 진입점 호출만 한다
- 룸 클리어 판정·웨이브 진행 — 1/4·3/4 소관. 문은 클리어 신호를 구독만 한다
- 분기 스테이지 선택 로직 (목적지 식별자를 실어 보내는 계약만 만들고, 여러 문 중 무엇을 고를지에 대한 규칙은 만들지 않는다)
- 이번 범위의 실제 배치는 룸당 문 1개. 다중 문 씬 저작은 하지 않는다
- 문 열림 연출(애니메이션·사운드·파티클), 잠긴 문·열쇠·조건부 문
- 프롬프트 텍스트 현지화, 게임패드 버튼 아이콘 전환
- 베이스 씬 HUD를 경유하는 상호작용 UI 시스템
- 자동화 테스트 — 테스트 어셈블리 도입 전까지 검증 안 함

## Technical Requirements

### 배치

- 문 View와 문 상호작용 계약 → `Assets/20_ProejctB/01_Scripts/Room/` (폴더명 `20_ProejctB`는 현재 리포지토리의 실제 철자)
- 문은 룸 씬(additive)에 배치되는 오브젝트다. 베이스 씬이 문을 탐색하지 않는다
- 공통 코어(`00_CommonFramework`)에 문·룸 구체 참조가 역류하지 않게 한다. 문이 쓰는 공통 자산은 `O2un.Input.IInputReader` 하나다

### 문 View (MonoBehaviour)

- 필수 컴포넌트: `Collider2D`(`isTrigger = true`). `[RequireComponent(typeof(Collider2D))]`로 강제한다
- `OnTriggerEnter2D` / `OnTriggerExit2D`로 플레이어 범위 상태를 갱신한다
- 플레이어 판별은 태그 문자열 비교가 아니라 **`LayerMask` 필드로 지정한 레이어 검사**로 한다. 씬 탐색(`GameObject.Find` 등) 금지
- 인스펙터 직렬화 필드: 목적지 식별자(`string`), 프롬프트 루트(`GameObject`), 플레이어 레이어 마스크
- 보유 상태는 두 가지뿐이다: 열림 여부, 플레이어 범위 내 여부. 이 두 값의 AND가 프롬프트 표시 조건이자 입력 수용 조건이다
- 진행 가능 여부·룸 인덱스·다음 룸 주소를 문이 판단하거나 보관하지 않는다

### 열림 / 닫힘 상태

- 초기 상태는 **닫힘**이다. 닫힘 상태에서는 프롬프트를 표시하지 않고, 범위 안이라도 입력을 신호로 바꾸지 않는다
- 열림 전환은 3/4의 `IRoomProgression.OnRoomCleared` 구독으로 받는다. 문이 잔존 적 수를 세거나 스포너를 직접 보지 않는다
- 트리거 콜라이더 자체는 닫힘 상태에서도 켜둔 채 상태 플래그로 막는다. 콜라이더를 껐다 켜면 이미 범위 안에 서 있던 플레이어의 `OnTriggerEnter2D`가 다시 오지 않아 프롬프트가 뜨지 않는다
- 따라서 **열림 전환 시점에 이미 범위 안이면 즉시 프롬프트를 표시한다**

### 프롬프트 UI

- 문 View가 소유한 **월드 스페이스** UI다. 문 오브젝트의 자식으로 두며, 베이스 씬 HUD 캔버스를 경유하지 않는다
- 캔버스 `renderMode = WorldSpace`. 룸 씬과 함께 로드·언로드되므로 베이스 씬에 잔재가 남지 않는다
- 표시 조건: 열림 && 범위 안. 이탈하면 숨긴다. 표시·숨김은 프롬프트 루트 `GameObject`의 활성 토글로 한다
- 룸 전환이 시작된 뒤에도 프롬프트가 남지 않도록, 전환 신호를 발행한 시점에 프롬프트를 숨긴다

### 입력

- 공통 `O2un.Input.IInputReader`의 `Observable<Unit> IsAttackPressed`를 구독한다. `UnityEngine.Input` 직접 폴링 금지
- `IInputReader`는 VContainer로 주입받는다. 문은 룸 씬 오브젝트이므로 룸 씬 `LifetimeScope`를 통해 부모(베이스 씬) 스코프에서 해소된다. `Container.Resolve<>()` 직접 호출 금지
- 구독은 상시 유지하고 **핸들러 안에서 상태를 검사해** 열림 && 범위 안이 아니면 아무 것도 하지 않는다. 범위 진입/이탈마다 구독을 붙였다 떼지 않는다 (이탈 타이밍과 입력 프레임이 엇갈릴 때 동작이 흔들린다)
- 구독은 `CompositeDisposable`에 `AddTo`하고 `OnDestroy`에서 해제한다

### 전환 신호

- 문은 C# `event`가 아니라 R3 `Subject`/`Observable`로 신호를 노출한다
- 신호에는 문 자신의 목적지 식별자를 실어 보낸다. 수신 측이 문 인스턴스를 구별할 수 있어야 한다

```csharp
public interface IRoomDoor
{
    Observable<string> OnTransitionRequested { get; }
}
```

- 문 → `IRoomProgression` 연결은 문이 직접 `RequestTransition`을 호출하거나, 룸 씬 스코프가 구독해 중계하거나 둘 중 하나다. 어느 쪽이든 **문은 `IRoomProgression`의 상태를 읽지 않는다** (전환 가능 여부 검사는 3/4가 이미 수행한다)
- 문 개수를 가정하지 않는다. 룸 씬 스코프는 자기 룸의 문 목록을 인스펙터 참조로 갖고, 각 문의 신호를 동일하게 중계한다
- 전환 중 중복 입력 무시는 3/4 `RequestTransition`의 상태 검사에 맡긴다. 문 쪽에 별도 쿨다운·연타 방지를 넣지 않는다

### 로직 분리

- 프롬프트 표시 여부·입력 수용 여부를 결정하는 판정은 `Collider2D`/`GameObject` 조작과 분리해 작성한다. View는 트리거 콜백과 활성 토글만 담당하고, 상태 조합 판정은 Unity API 비의존으로 둔다
- 판정이 열림·범위 두 플래그의 AND뿐이므로 별도 Module 파일로 쪼갤 필요는 없다. 다만 씬·오브젝트 조작 코드와 섞어 쓰지 않는다

## Acceptance Criteria

- [ ] 룸 클리어 전에는 문 범위에 들어가도 프롬프트가 뜨지 않는다
- [ ] 룸 클리어 전에 문 범위 안에서 공격 키를 눌러도 룸이 넘어가지 않는다
- [ ] 룸 클리어 후 문 범위에 들어가면 문 위치에 월드 스페이스 프롬프트가 뜬다
- [ ] 범위에서 벗어나면 프롬프트가 사라진다
- [ ] 룸 클리어 시점에 이미 문 범위 안에 서 있으면 그 즉시 프롬프트가 뜬다
- [ ] 프롬프트가 뜬 상태에서 공격 키를 누르면 전환이 시작된다
- [ ] 문 범위 밖에서 공격 키를 눌러도 전환이 시작되지 않는다 (평소 공격 동작에는 영향이 없다)
- [ ] 전환이 시작되면 프롬프트가 즉시 사라진다
- [ ] 문 신호에 그 문의 목적지 식별자가 실려 전달된다
- [ ] 한 룸에 문을 두 개 배치해도 각각 자기 식별자로 신호를 보내고 서로 간섭하지 않는다
- [ ] 문 범위 안에서 공격 키를 연타해도 룸이 한 칸만 넘어간다
- [ ] 룸 씬이 언로드되면 문과 프롬프트가 함께 사라지고 베이스 씬에 잔재가 남지 않는다
- [ ] 룸 씬 안의 문이 `IInputReader`를 주입받아 동작한다 (주입 실패 예외 없음)
- [ ] Play 모드 진행 중 콘솔에 에러가 없다

## Open Questions

- 플레이어 루트 오브젝트의 실제 레이어 — 프로젝트에 `Player`(31)와 `Actor`(10) 레이어가 모두 존재한다. 문의 레이어 마스크 기본값을 확정하려면 프리팹 확인 필요
- 공격 키가 곧 상호작용 키이므로, 문 범위 안에서 누른 입력이 플레이어의 근접 공격도 함께 발동시킬지 (현재는 문 신호와 공격이 동시에 나가는 것을 허용한다고 전제)
- 프롬프트 표기 방식 — 텍스트("공격 키로 이동")인지 키 아이콘인지
- 문 → `IRoomProgression` 중계를 문이 직접 호출로 할지 룸 씬 스코프 중계로 할지 (설계 단계에서 확정)
- 룸 클리어 후 문이 열릴 때의 시각 피드백 유무 — 현재는 프롬프트 외 연출 없음
