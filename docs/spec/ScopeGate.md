# ScopeGate 스킬 — 요구사항 문서

> 하네스에 추가할 **요청 규모 판단 스킬**(`scope-gate`)의 요구사항 문서.
> 이 문서는 실제 프로젝트에서 요구사항 문서 워크플로(`/prd` 등)로 재생성해 정상 산출되는지 확인하기 위한 참조본이다.
> 근거 사양(Claude Code Skills, 2026-07 code.claude.com/docs 확인): 스킬은 `.claude/skills/<name>/SKILL.md`에 두며 슬래시 호출명은 폴더명에서 온다. frontmatter로 자동 호출 여부를 제어(`disable-model-invocation`)하고, 인자는 `argument-hint`/`arguments`/`$ARGUMENTS`를 사용한다.

## Overview

큰 작업을 한 덩어리로 설계·구현하면 결과물은 나오더라도 한 번에 맡긴 범위가 커져 토큰·시간 비용이 커진다. 사람의 규모 눈대중은 실제보다 작게 잡히기도 한다. `scope-gate`는 설계·구현에 **들어가기 전에** 요청(또는 이미 작성된 PRD)의 크기를 판단해, 크면 착수를 막고 **의존성 순서가 흐르는 PRD 분해안**을 제시하는 스킬이다. 사람의 초기 판단이 틀려도 착수 전 한 번 더 걸러 "큰 걸 통째로 태우는" 비용 사고를 방지한다.

## Goals

- 설계 착수 전, 요청 규모를 **판단 기준**으로 평가해 "한 번에 진행 / 분리 필요" 중 하나로 판정한다.
- 자동·수동 **두 갈래**로 호출된다.
  - **자동**: `unity-dev-orchestrator` 진입 시 설계 착수 전에 자동으로 규모를 판단한다.
  - **수동**: 사용자가 `/scope-gate`로 직접 호출한다. 인자로 **기존 PRD 파일 경로**를 받으면 그 PRD가 적정 크기인지, 나눴어야 하는지 검증한다.
- "분리 필요" 판정 시 **의존성 순서가 흐르는 PRD 후보 목록**을 제시한다. 각 후보는 하나의 모듈 경계에 대응하고, **입력(Depends On)**·**출력(Provides)**을 한 줄씩 명시해 앞 후보의 출력이 다음 후보의 입력이 되는 사슬을 이룬다.
- 판정만 하고 **자동으로 작업을 시작하지 않는다** — 사용자 승인 뒤에 후속(PRD 하나씩 생성, 파이프라인)으로 넘어간다.

## Out of Scope

- PRD 파일 자체의 생성·저장. 분해안 확정 후 실제 PRD 작성은 기존 `prd` 스킬(분리 모드)이 담당한다.
- 코드 구현·씬 조작·4단계 검증(컴파일/플레이/기능 테스트). 이 스킬은 착수 전 판단·분해 제안까지만 한다.
- 규모가 "작다"고 판정된 요청의 처리. 이 경우 게이트는 통과만 시키고 기존 흐름에 맡긴다.
- 비용·토큰의 정량 측정. 파일 수·시스템 수 등 구조적 신호로만 판단한다(실측 토큰 계산 아님).

## Technical Requirements

### 스킬 배치·frontmatter

- 위치: `.claude/skills/scope-gate/SKILL.md`.
- 호출명: 폴더명 `scope-gate`에서 `/scope-gate`가 된다.
- frontmatter:
  - `name: scope-gate`(표시명)
  - `description`: "요청/PRD의 규모를 설계 착수 전에 판단해 크면 의존성 순서 PRD 분해안을 제시한다" 취지 + 자동 트리거 문구.
  - `disable-model-invocation`을 **설정하지 않는다(기본값 false)** — 자동(모델 트리거)·수동(`/scope-gate`) 둘 다 가능해야 하므로. (`true`로 두면 자동 호출이 막힌다.)
  - `argument-hint: "[PRD 경로] (생략 시 현재 요청 규모 판단)"`.
  - `allowed-tools`: 읽기·검색 위주(`Read`, `Grep`, `Glob`). 파일 생성·수정은 하지 않는다.
- 인자 처리: `$ARGUMENTS`가 PRD 경로면 해당 파일을 읽어 검증 모드로, 비어 있으면 현재 대화의 요청을 판단 대상으로 삼는다.

### 판단 기준 ("큰 요청" 신호 — 2개 이상이면 분리 필요로 본다)

- **여러 시스템에 걸침**: 독립적으로 검증 가능한 시스템/기능이 2개 이상 얽혀 있다.
- **다단계 의존 흐름**: 한 부분의 산출물(이벤트·인터페이스·데이터)이 다음 부분의 입력이 되는 순차 의존이 존재한다.
- **레이어 다수 신설**: Manager/Module/Service/View 계열을 여러 벌 새로 만든다.
- **신규 파일 다수**: 새로 생길 `.cs`가 대략 5개를 넘길 것으로 보인다.
- **요구가 3마디 이상**: "A 하고 B 하고 C까지" 형태로 이어진다.

### 출력 계약

- 판정 결과: `한 번에 진행` 또는 `분리 필요`.
- `분리 필요`일 때 분해안 표: 각 행이 PRD 후보 1건.

  | 순서 | PRD 후보 | 입력(Depends On) | 출력(Provides) |
  |---|---|---|---|
  | 1 | … | (선행 조건) | (다음이 받을 산출) |
  | 2 | … | 1의 출력 | … |

- 후속 안내: "이 경계로 진행할까요, 조정할까요?"를 묻고 승인 전 착수 금지. 승인 시 `prd` 분리 모드로 PRD를 하나씩 생성하도록 넘긴다.

### 수동 검증 모드 (인자에 PRD 경로)

- 대상 PRD를 읽어 그 안에 독립 검증 가능한 시스템이 몇 개 섞였는지, 판단 기준에 몇 개 걸리는지 리포트한다.
- 하나로 뭉쳐 있고 기준 2개 이상에 걸리면 "N개로 나누는 것을 권장" + 위 분해안 표를 제시한다.

## Acceptance Criteria

- [ ] `/scope-gate` 수동 호출과 orchestrator 진입 시 자동 호출이 **둘 다** 동작한다(`disable-model-invocation`이 켜져 있지 않다).
- [ ] 인자 없이 호출하면 현재 요청을, PRD 경로를 주면 그 파일을 판단 대상으로 삼는다.
- [ ] 판단 기준 2개 이상에 걸리면 `분리 필요`로 판정하고, 걸린 기준을 근거로 함께 제시한다.
- [ ] `분리 필요` 시 입력·출력이 사슬로 이어지는 PRD 후보 표를 제시한다(앞 출력 = 다음 입력).
- [ ] 판정 후 **자동으로 설계·구현·PRD 생성을 시작하지 않고**, 사용자 승인을 기다린다.
- [ ] 수동 검증 모드에서 단일 PRD 파일(예: 베이스+스킬 3종을 한 파일에 담은 PRD)을 주면 "나누는 것을 권장"과 분해안을 반환한다.
- [ ] 이 스킬은 파일을 생성·수정하지 않는다(읽기·판단·제안까지만).

## Open Questions

- 스킬 이름을 `scope-gate`로 할지 `request-sizer` 등 다른 이름으로 할지.
- "신규 파일 다수"의 임계치(대략 5개)를 프로젝트 규모에 맞게 조정할지.
- 분해안 제시 후 승인·PRD 생성 연결을 orchestrator가 맡을지, 이 스킬이 `prd` 분리 모드를 직접 호출할지.
