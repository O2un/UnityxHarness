# 2DPlayerMovement — /prd 입력 답변

> `/prd 2DPlayerMovement` 실행 후 이어지는 질문에 붙여넣을 답변 정리. 근거 설계: 4B-2c1 대본(Rigidbody2D 물리 이동 + 조작감 보정 3종 + 카메라 보정).

| 질문 | 답변 |
|---|---|
| 무엇을 만드는가 | 2D 플랫폼 액션의 플레이어 이동. 좌우 이동과 점프를 Rigidbody2D 물리로 처리하고, 그 위에 코요테 타임·점프 버퍼·관성 보정을 얹어 조작감을 값으로 튜닝할 수 있게 한다. 카메라의 데드존·룩어헤드까지 이동 조작감에 포함한다 |
| 구성요소 | `PlayerMover`(속도 계산·코요테/버퍼 판정·가감속 램프, 순수 C# Module), 입력 리더(좌우는 Reactive 값, 점프는 눌림 이벤트), `PlayerView`(Rigidbody2D 구동·접지 판정 등 Unity 표현), 카메라는 Part 2 카메라 계층 위 Cinemachine Position Composer의 Dead Zone·Lookahead |
| 물리 이동 방식 | Transform으로 좌표를 직접 더하지 않고 Rigidbody2D에 속도를 준다. 벽·발판·경사 충돌과 중력은 물리 엔진에 맡기고 우리는 속도만 정한다. 좌우 목표 속도·점프 판정은 Tick(Update) 시점에 계산하고, Rigidbody2D 속도 적용은 FixedUpdate에서 한다 — 프레임률이 흔들려도 이동이 일정하도록 |
| 입력·점프 계약 | 좌우 입력은 Reactive 값으로 받아 Mover가 구독한다. 점프 눌림은 이벤트로 들어오면 기록해 뒀다가 다음 물리 스텝에서 딱 한 번만 소비한다 — FixedUpdate가 한 프레임에 0·1·N회 돌 수 있으므로, 이 "기록 후 한 번 소비" 규칙이 점프가 정확히 한 번 나가게 하는 기준이다 |
| 보정 3종 | 코요테 타임(발판을 벗어난 직후 짧은 유예 동안 점프 입력을 받아줌) / 점프 버퍼(착지 직전에 누른 점프를 기억했다가 착지 순간 발동 — 점프 계약 위에서 동작) / 관성(가속·감속에 램프를 줘 무게감). 셋 다 코드에 숨기지 않고 조절 가능한 값으로 둔다 |
| 카메라 보정 | 데드존(캐릭터가 화면 중앙 작은 영역 안에서 움직이는 동안 카메라가 따라오지 않아 잔떨림을 줄임) / 룩어헤드(진행·낙하 방향으로 화면을 살짝 미리 밀어 앞을 먼저 보여줌). 2D·직교 카메라에서 Position Composer가 이 값을 제공하고, 실제 적용은 Part 2에서 잡은 카메라 계층이 맡는다 |
| 역할 분리 | 언제 얼마의 속도로 갈지, 코요테 유예가 남았는지 같은 판단은 Unity API에 의존하지 않는 순수 `PlayerMover`가 소유한다. Rigidbody2D에 속도를 넣고 접지를 감지하는 구동은 `PlayerView`가 맡는다. 카메라 값 적용은 카메라 계층 — 세 경계가 서로의 구현에 의존하지 않는다 |
| 데이터 | 코요테 타임·점프 버퍼 시간, 가감속 램프, 최고 속도·점프 힘, 카메라 Dead Zone·Lookahead 크기는 코드 상수가 아니라 조절 가능한 값(예: `ScriptableObject`)으로 두고 Context가 입력받아 Mover·카메라 계층에 넘긴다 |
| 배치 | 이 2D 이동은 프로젝트 B에서만 쓰는 로직이므로 코드는 `Assets/20_ProjectB` 쪽에 둔다. 공통 코어(`00_CommonFramework`)의 Actor/Context/View/Module 경계는 그대로 따른다 |
| 이번 범위 | 좌우 이동·점프·보정 3종·카메라 데드존/룩어헤드까지. 대시·벽점프·이중 점프, 공격·피격은 범위 밖(뒤 클립) |
