# RoomSystem — /prd 입력 답변

> `/prd 룸 전환 & 스테이지 구조` 실행 후 이어지는 질문에 붙여넣을 답변 정리. 근거 설계: 4B-2c7 대본(베이스 씬 유지 + 룸만 additive, 스폰→클리어→문→화면 전환 사이클) + 공통 코어 실물 코드(`Labinno.Core.SceneManager`의 `IAdditiveSceneLoader` — `LoadAdditiveSceneAsync(key, parentScope)`에서 `LifetimeScope.EnqueueParent`로 부모 스코프 주입, `Addressables.LoadSceneAsync(Additive)`), 프로젝트 A의 EnemySpawner/WaveModule.

| 질문 | 답변 |
|---|---|
| 무엇을 만드는가 | 2D 플랫폼 스테이지를 여러 룸으로 나눠 잇는 룸 전환 시스템. 플레이어·카메라·공통 시스템이 담긴 **베이스 씬은 로드된 채 유지**하고, 방마다 바뀌는 맵·스폰 배치만 **additive 룸 씬**으로 얹고 내린다. 룸 사이클은 스폰 → 클리어 판정 → 문 트리거 → 화면 전환 → 다음 룸 |
| 구성요소 | `RoomModule`(순수 C# 진행 로직 — 다음 룸 결정·스폰 요청·클리어 판정·전환 오케스트레이션), 룸 데이터(각 룸의 additive 씬 주소 + 스폰 정보 보유), 문 상호작용 View(2D 트리거 콜라이더 + 상호작용 프롬프트 UI — 범위 진입 시 프롬프트 표시, 공격 키 입력 시 R3 이벤트), 화면 전환 View(페이드 연출). 씬 로드·언로드·수동 스코프 주입은 공통 SceneService에 위임 |
| 재사용 vs 전용 경로 | **재사용(공통, 그대로 참조)**: SceneService(`ISceneService`/`IAdditiveSceneLoader` — additive 로드·언로드와 부모 스코프 수동 주입), 프로젝트 A의 EnemySpawner/WaveModule(스폰·웨이브 규칙), Addressables 주소 로드, VContainer LifetimeScope 계층. **Project B 전용으로 추가**: `RoomModule`, 룸 데이터(씬 주소+스폰 정보), 문 상호작용 View(트리거 + 프롬프트 UI), 화면 전환(페이드) View |
| 씬 구성 | 베이스 씬 1개(플레이어·카메라·HUD·공통 시스템, 게임 내내 유지) + 룸 씬 N개(맵 지오메트리 + 스폰 정보만, additive로 얹고 내림). 룸을 넘어갈 때 실제로 오르내리는 건 룸 씬뿐이고 베이스는 재로드하지 않는다 |
| 룸의 소스 | 이번 프로젝트는 룸을 **미리 저작한 additive 씬**으로 채운다(맵·스폰 배치를 씬에 담아 주소로 로드). `RoomModule`은 "다음 룸이 무엇인지"만 알고 룸을 실제로 채우는 방식은 소스에 맡긴다 — 절차적 지형 생성 같은 다른 소스도 "베이스 위에 얹고 다 지나면 내린다"는 같은 형태로 끼운다(이번 범위에서 절차적 생성 자체는 구현하지 않음) |
| additive 로드와 수동 주입 | 룸 씬 로드·언로드는 공통 SceneService가 처리한다: 다음 룸을 주소로 additive 로드하고 이전 룸은 언로드. **런타임에 additive로 올라온 씬은 자기 DI 컨테이너를 가지므로**, 그 씬 안 오브젝트가 베이스 씬의 공통 시스템을 주입받으려면 로드 시 **부모 스코프를 수동으로 넘겨야** 한다(`LifetimeScope.EnqueueParent(parentScope)` 흐름). 이 주입은 SceneService의 additive 로드 안에서 함께 이뤄진다. `RoomModule`은 SceneService 계약에 의존해 "이 룸을 올려 달라"고 요청할 뿐, 로드·주입 내부는 모른다 |
| 스폰 & 클리어 | 룸이 올라오면 그 룸의 스폰 정보를 프로젝트 A의 스포너에 넘겨 적을 세운다(스폰·웨이브 규칙은 재사용). 이 룸에서 스폰된 적이 **전부 처치되면 클리어**로 판정하고, 다음 룸으로 가는 문이 열린다 |
| 문 상호작용 & 화면 전환 | 문은 **View의 2D 트리거 콜라이더 + 상호작용 프롬프트 UI** — 클리어로 열린 뒤 플레이어가 문 범위에 들어오면 **프롬프트 UI를 띄우고(예: "X"), 공격 키를 누를 때** R3 이벤트로 `RoomModule`에 알린다(범위를 벗어나면 프롬프트 숨김). 입력 판정·프롬프트 표시는 View가 맡고 `RoomModule`에는 신호만 전달한다. 신호를 받으면 전환: **페이드 아웃 → 현재 룸 언로드 → 다음 룸 로드 → 페이드 인 → 스폰** 순서. 프롬프트 UI·페이드는 View가 소유한다(페이드는 로드 순간을 가리는 연출, single 로드용 로딩 화면과 별개) |
| 스테이지 구조 | 스테이지는 룸들의 **시퀀스**다. 문을 지날 때마다 다음 룸으로 넘어가고, 마지막 룸까지 정리하면 스테이지 클리어. 룸 순서·구성은 데이터로 두어 코드 수정 없이 스테이지를 엮는다 |
| 역할 분리 | `RoomModule`은 **순수 진행 로직** — 씬을 직접 다루거나 트리거를 읽지 않고, SceneService·스포너·문 이벤트를 **주입받아 호출만** 한다. 그래서 "클리어되면 다음 룸으로 넘어가는가", "마지막 룸이면 스테이지 클리어인가"를 Unity 없이 `new`로 만들어 Play 없이 테스트로 확인한다. 씬 로드·언로드·수동 주입은 **공통 SceneService**, 문 트리거·프롬프트 UI·입력 판정·페이드 표현은 **View**가 맡고 `RoomModule`에는 이벤트로만 전달한다 |
| 데이터 | 룸 순서·각 룸의 씬 주소·스폰 정보, 페이드 시간은 코드 상수가 아니라 조절 가능한 값(`ScriptableObject`/룸 데이터)으로 두고 Context가 입력받아 `RoomModule`에 넘긴다 |
| 배치 | `RoomModule`·룸 데이터·문 상호작용 View(트리거+프롬프트 UI)·화면 전환 View는 프로젝트 B에서만 쓰는 로직이므로 `Assets/20_ProjectB`에 둔다. 공통 코어(`00_CommonFramework`)의 SceneService·스포너·웨이브·Addressables는 참조만 한다(재작성하지 않음). 공통 베이스에 특정 룸·씬 이름 같은 구체 참조가 역류하지 않게 한다 |
| 이번 범위 | 룸의 씬 구성(베이스+additive)·소스 추상·수동 주입·스폰/클리어·문 상호작용(트리거+프롬프트+공격 키 입력)·화면 전환·스테이지 시퀀스 **설계**까지. 실제 `RoomModule`·문 상호작용 View·페이드 구현과 룸 간 데이터 인계는 다음 구현 클립에서. 룸 내부 세이브/체크포인트, 미니맵, 분기 스테이지는 범위 밖(뒤 클립) |
