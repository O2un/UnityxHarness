**$ARGUMENTS 게임**의 기획 문서를 작성해 `docs/design/game-plan.md`에 저장한다.

이 스킬의 목적은 게임의 핵심 루프와 기능 우선순위를 정리해, 이후 구현을 어떤 순서로 시작할지 판단할 수 있는 기획 문서를 만드는 것이다. 가장 중요한 결과물은 **무엇을 먼저 만들지(MVP)** 와 **어떤 순서로 만들지(개발 순서)** 다.

## 입력

사용자에게서 다음 세 가지를 받는다. 인자나 대화에 이미 있으면 그대로 쓰고, 빠진 것만 질문한다.

- **장르** (예: 로그라이크, 타워 디펜스, 방치형)
- **핵심 루프** (플레이어가 반복하는 한 사이클)
- **주요 시스템 목록** (예: 전투, 인벤토리, 상점, 성장)

## 동작 절차

1. **입력 파악** — 장르와 핵심 루프를 이해한다. 빠진 입력은 질문한다.
2. **레퍼런스 후보 검색** — 같은 장르라도 대표작마다 루프·시스템 구성이 다르므로, 웹 검색으로 유사 장르 게임을 찾아 **후보 3~5개**를 제시한다. 각 후보에는 핵심 루프·차별점을 한 줄로 붙인다.
3. **레퍼런스 선정 질문** — 어떤 게임을 레퍼런스로 삼을지 사용자에게 되묻는다. 임의로 확정하지 않는다. 선정된 게임의 루프·시스템 구성을 이후 기획에 반영한다.
4. **핵심 루프 정리** — 플레이어가 반복하는 사이클을 단계별 순서도(텍스트)로 정리한다.
5. **주요 시스템 정리** — 각 시스템을 역할과 함께 정리한다. 이 단위가 이후 구현의 클립 단위가 된다.
6. **기능 우선순위 분할** — 기능을 **MVP**와 **나중**으로 나눈다. MVP 기준은 사용자와 합의한다. 임의로 범위를 확정하지 않는다.
7. **개발 순서 제안** — 구현할 순서를 번호로 제안한다. 의존 관계상 먼저 있어야 하는 것을 앞에 둔다.
8. **리뷰 후 저장** — 초안을 사용자에게 보여주고 검토를 요청한다. 승인되면 `docs/design/game-plan.md`에 저장한다. 확정 전까지 파일을 저장하지 않는다.

## 출력 문서 구조

`docs/design/game-plan.md`는 아래 순서를 그대로 따른다.

| 섹션 | 내용 |
|------|------|
| Game Concept | 장르의 핵심 경험을 한 단락으로 |
| Reference Games | 사용자가 선정한 레퍼런스 게임과 각 게임에서 참고할 요소 |
| Core Loop | 각 단계에서 무엇이 일어나는지 순서도로 |
| Key Systems | 시스템별 역할 (구현 클립의 단위) |
| Feature Priority | MVP / 나중 2분할. **이 문서에서 가장 중요한 섹션** |
| Development Order | 구현 순서를 번호로 |

## 작성 규칙

- 파일은 **반드시** `docs/design/` 폴더에 저장한다. 폴더가 없으면 생성한다. 다른 위치 금지.
- 문서는 **한국어**로 작성한다.
- 레퍼런스 게임은 임의로 확정하지 말고, 웹 검색으로 후보를 제시한 뒤 사용자에게 선정받는다. 검색 없이 기억에만 의존해 후보를 지어내지 않는다.
- MVP 범위는 사용자와 합의한다. 어디까지가 MVP인지 애매하면 확정하지 말고 사용자에게 확인한다. 잘못 잡은 MVP는 잘못된 개발 순서로 이어진다.
- 알 수 없는 값은 추측하지 않고 질문한다.
- 이미 `docs/design/game-plan.md`가 존재하면 덮어쓰기 전에 사용자에게 확인한다.
